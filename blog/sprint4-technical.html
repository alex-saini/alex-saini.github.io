KATA, Problem Solving & Code Readability

A time I got blocked on a simple problem.

Not that much of a simple problem, but a simple solution to the problem... I was doing the JS Gradebook exercise within the Kata and couldn't seem to debug why my code wasn't working. I had been doing the work for a few hours without a break so my mind was relatively fuzzy at this point.

How did I solve it? What techniques did I use and what I did I learn?

I first went through the code and tried to properly decipher what was wrong, went to the internet straight after, and after a hour of tedious research I went to seek help from a tutor. I gave Ollie a call, and he stepped me through the process of what I had done wrong, and it seemed to be a notation problem. Even though I had used bracket notation I hadn't placed a '.' after so it couldn't access the next property. Guess it shows taking a break every once a while instead of bulking out work is a good idea. Also learn't about the importance of console logging, the way that Ollie was console logging to check what was wrong with peices of code showed me how much I was underrating the problem solving technique.

Reflection on problem solving techniques:

  • Pseudocode: Confident, use it quite regularly on and off the work.
  • Trying something: Do it all the time, very confident.
  • Rubber ducky method: Don't use it.
  • Reading error messages: Do this pretty consistently, confident about using it.
  • Console logging: Wasn't too confident before this sprint, but am very confident with console logging now.
  • Googling: Very confident.
  • Asking your peers for help: Not that confident using this, need to do push myself to ask for help more.
  • Asking coaches for help: Confident with doing so if I need too.
  • Improving your process with reflection: Confident at reflecting.
